Zero2x 终极融合版：AI-Native 渐进式科研工作台 (AI-Native Progressive Scientific Workbench)

核心设计哲学：渐进式揭示 (Progressive Disclosure)

我们将 Zero2x 的体验设计为一场**“从仰望星空到动手创造”**的旅程。

前半程（引人入胜）：采用 2.0 方案的电影级滚动叙事（Scrollytelling）。页面轻量、视觉震撼，向用户全景展示 Zero2x 的模型能力与数据底座。

后半程（硬核赋能）：在叙事的终点，自然过渡到 3.0 方案的“AI-Native 科研工作台”。告诉科学家：“您刚才看到的所有奇迹，现在只需一行指令即可亲手复现。”

## 0.1 目标与现状差距（更新于 2026-03-03）

本节用于把 v4 的“体验目标”落到本仓库现状，并明确下一步要补的内容。

当前已实现（工程边界正确）：

- 路由分离：`/`（Landing 轻叙事）/ `/act2`（第二幕叙事 Cesium）/ `/demo`（验证系统）/ `/workbench`（工作台）
- 第二幕已具备电影化骨架：滚动分镜、信息卡片、多目标切换、推荐路径 tours，并且通过测试门禁保持与 `/demo` 解耦
- 第四幕已切换为“全视频流”预告（Landing 不再默认挂载 WebGL 星海）；第五幕升级为“叙事即工具”的融合工作台预览（散点式 HUD / 去容器化 + 中央负空间 + 物理居中准星），并保留 `/workbench` 作为重应用（Monaco 按需加载；Act5 点击时可携带上下文如 `/workbench?context=poyang`）

新增关键结论（来自 update_patch_0303 的截图复盘）：

- **布局稳健性是第一优先级**：必须在 75% 缩放 / 高分屏 / 超宽屏下维持“背景视频的视觉中心不被遮挡”。
- **去容器化 (De-containment)**：Act5 不再使用大面积封闭窗体；改用散点式 HUD 把信息分布在视口边缘，强制中间留白。

主要差距（需要补“视觉与叙事”）：

1) 第一幕视觉仍偏“工程占位”
	- v4 目标是“呼吸态发光神经网络球体 + 极简 Omni‑Bar + 高级感氛围”
	- 现状虽有深色背景与 Omni‑Bar，但缺少可复用的“主视觉资产”（球体/星尘/微动效），导致品牌冲击不足

2) Landing 的“向下滑动叙事”仍需更电影
	- v4 目标是滚动驱动的镜头/卡片/渐进揭示（可 GSAP，也可 IO 占位）
	- 现状虽已具备滚动分幕结构，但需要更明确的“滚动反馈”（进度/高亮/显隐动效/镜头替身）

3) `/demo` 入口需要降噪
	- v4 语义：`/demo` 是“验证系统/工具台”，不应抢占首屏转化与叙事节奏
	- 策略：把 Demo 入口收敛为少量位置（页脚/Act5/高级入口），让默认路径走“叙事 → 工作台”

4) Act4/Act5 的“电影级素材资产”需要投放
	- Act4：`act4_galaxy.webm/mp4`（语义星海预告片，极缓慢镜头 + Bloom/光流）
	- Act5：`act5_workbench_bg.webm/mp4`（指挥舱/轨道实验室窗外视界，低对比度，适配前景代码可读性）
	- 目标：在不增加前端逻辑复杂度的前提下，把“视觉统一性”和“沉浸感连续性”拉满

5) 缩放/移动端布局稳健性需要制度化（验收而非主观）
	- 典型场景：浏览器缩放 75%，或者高分屏导致“视口容纳信息密度变高”
	- 风险：固定高度/固定偏移（例如硬编码的 -vh 位移）会导致视觉重心漂移；内容块会在垂直空间压缩时“撑满画面”，遮挡背景主体
	- 策略：
		- 禁用影响缩放的负偏移（如 Act1 不能用 `margin-top: -10vh` 作为视觉对齐手段）
		- 所有关键字号/面板宽度使用 `clamp()` / `min()`
		- Act5 中央视窗必须强制留白（约 35%~45% 宽度）
		- Act5 准星使用物理居中（50%/50% + translate）确保在缩放/超宽屏下仍对准屏幕死中心

## 0. 工程落地约束（以本仓库为基底）

本仓库并不是从零搭建的官网，而是一个已经可运行的“Cesium 场景验证系统”。Zero2x 升级必须做到：

- **不推倒重来**：保留现有 AlphaEarth Cesium Demo 能力作为“工作台/场景验证”子应用。
- **渐进式揭示**：根路径 `/` 负责“轻量叙事 + Omni-Bar 引导”，重应用挂在 `/demo`（或其它固定子路径）。
- **性能优先**：Zero2x 首页不应默认初始化沉重的 Cesium Viewer；需要“按路由/按需加载”的分离。
- **可复现 + 可验收**：以 Docker Dev/Prod 为主路径，端口统一：Dev=8404/8405，Prod=8406/8407。

## 1. 技术栈（建议，吸收 v2/v3 思路但对齐仓库现实）

### 1.1 本仓库当前真实栈（作为基线）

- 前端：**Vue 3 + Vite + CesiumJS**
- 后端：**FastAPI + httpx + Google Earth Engine（可 stub）**
- 测试：后端 pytest；前端 vitest

### 1.2 Zero2x 首页（叙事页）建议栈

- 3D/粒子/微观动效：优先 **Three.js（原生）**（避免引入 React 生态）
- 滚动编排：优先 **GSAP + ScrollTrigger**（若 48h MVP 来不及，可先用 CSS/IntersectionObserver 占位）
- 文本/指令交互：Omni-Bar（全局 ⌘K / Ctrl+K），支持流式输出占位（Typewriter）

### 1.3 “AI-Native 工作台”建议栈（后半程）

- 代码编辑器：**Monaco Editor**（VS Code 内核）
- Notebook（可选进阶）：**JupyterLite**（浏览器内 WASM Python）或 iframe 嵌入外部 Notebook
- 流式对话：后端可用 **SSE**（EventSource）或 **chunked fetch**（ReadableStream）

> v2/v3 提到的 Next.js/Tailwind/Vercel AI SDK 属于“从 0 造站”的理想路线；本仓库更稳妥的路径是：
> 先在 Vue/Vite 内实现 Zero2x 首页与工作台壳，再按需引入 Tailwind/Monaco/GSAP。

🛠 实施路径与技术选型 (2天 MVP & 1周原型)

为了在一周内实现这个看似庞大的“融合方案”，前端工程必须做到**“重渲染，轻逻辑”**。

1. 首页性能优化策略 (Performance First)

分离架构：首页（第一到第五幕的展示页）绝对不加载 Monaco Editor (VS Code 内核) 和 Jupyter 等重型组件。

按需加载：首页只加载必要的 Three.js (处理星空、DNA、粒子星系) 和 CesiumJS (处理数字地球)，使用 GSAP 统一管理所有的滚动动画。

视频/动图降级：在 2 天的 MVP 概念版本中，如果微观材料（OnePorous）的 3D 生成动画太难写，可以直接用一段高质量的 3D 渲染视频或 WebP 动图代替，配合 GSAP 控制视频帧播放，能达到以假乱真的滚动交互效果。

2. MVP 概念版核心里程碑（48小时）

搭建骨架：完成黑色主视觉，画出首屏 Omni-Bar 和底部 Workspace 概念静态 UI（可先用纯 CSS；后续再引入 Tailwind）。

打通“宇宙-地球”核心转场：重点死磕第二幕。引入 CesiumJS，实现从深空星轨到地球表面的平滑推镜效果。这是最能体现平台“高大上”和“全尺度”的 Killer Feature。

模拟 AI 生成：第五幕的工作台只需写死几段 CSS 动画，模拟代码自动生成（Typewriter 效果）和图表淡入，传达“Agent 自动生成”的理念即可，无需真实对接大模型 API。

3. 一周完整原型版升级（7天）

对接 021 问答：将首屏的 Omni-Bar 真正对接一个轻量级 LLM API。用户提问科研问题，可以流式输出解答，增强平台的“AI 属性”体感。

完善数据星海：使用 react-force-graph-3d 库，导入一批假的学术数据 JSON，生成可鼠标拖拽、悬停高亮的 3D 数据拓扑网，完成第四幕。

> 注：本仓库前端是 Vue。若不引入 React，可用 `3d-force-graph` / `three-forcegraph`（原生 three 包装）实现同等效果。

## 4. 工程化“造假策略”（允许的 MVP 捷径）

- **打字机/流式**：先用纯前端 Typewriter 模拟“流式生成代码/报告”，不必马上对接真实 LLM。
- **3D 复杂场景降级**：OnePorous / DNA 可先用高质量视频/WebP，滚动时“帧播放/淡入淡出”。
- **数据星海假数据**：先用 500–1000 节点 JSON（含 x/y/z + topic），保证交互“像真的”。
- **工作台先壳后芯**：先把三栏 IDE 结构做出来（Agent Flow / 3D Preview / Papers），再逐步塞 Monaco/Jupyter。
