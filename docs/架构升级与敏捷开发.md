AlphaEarth/Zero2x 架构升级与敏捷开发全景指南（基于仓库现状修订）

最后更新：2026-02-27

本指南的目标不是“为了 Docker 而 Docker”，而是以 可复现、可回滚、可观测、可 TDD 为约束，对当前工程做系统升级。

1. 现状复盘（以仓库真实实现为准）

1.1 当前端口与环境边界（事实）

仓库当前已经形成稳定的“开发/生产双端口隔离”策略：

| 环境 | 前端端口 | 后端端口 | 运行形态 | 关键机制 |
|---|---:|---:|---|---|
| Dev（默认 021） | 8404 | 8405 | Docker Compose（隔离容器） | Vite 反代 `/api` -> 8405；隔离的 `backend_test` / `frontend_test` 避免宿主污染 |
| Prod（Docker 版） | 8406 | 8407 | Nginx 静态 dist + FastAPI（Docker Compose） | 静态前端同源代理 `/api` -> 8407；更接近生产形态，便于本地复现 |
| Canary（可选） | 8508 | 8509 | 仅在确有需求时引入 | 用于灰度验证，不应替代现行稳定 Prod 方案 |

重要一致性：仓库已经提供 /api/debug/version 用于确认运行中的 git sha、python 可执行路径与依赖版本，从根上解决“看起来改了代码但线上还在跑旧进程”的排障难题。

补充：前端 Cesium 渲染链路已固化“可解释/可控”的视觉策略（避免宏观视角马赛克、业务图层遮挡等），相关开关已在 `.env.example` 中给出。

1.2 当前 CI/CD（事实）

CI：.github/workflows/ci.yml（push/PR 触发）

后端：make test-fast（默认 stubs 路径）

前端：npm ci && npm test && npm run build

上传 frontend/dist artifact

生产发布：.github/workflows/deploy_prod.yml（workflow_dispatch 手动触发）

先跑测试，再构建 dist

打包 release tar.gz 上传服务器

服务器侧 deploy/scripts/deploy_release.sh 切换 current 软链并重启 systemd，可自动健康检查与回滚

这套机制的优点是：发布可控（不误上线）、可追溯（release_id + sha）、可回滚（软链切换）。

2. 升级路线反思与纠偏

原版本把“Docker 作为唯一目标”放得过于靠前，导致与仓库现状出现多处不一致。我们需要纠偏：

端口事实：本仓库 Dev 的 Vite 端口固定为 8404；后端 Dev 默认 8405；Prod（Docker 版）为 8406/8407。

变量命名：前端真实使用的是 VITE_API_BASE，不是 VITE_API_BASE_URL。

稳妥优先：Docker 优先解决本地 Dev/Prod 一致性问题，随后在 8508/8509 建立 Canary 环境进行验证。

3. 升级目标（可验收的工程目标）

3.1 稳定性与可观测

所有关键链路都能用一个“自检面板”快速定位：

GET /health

GET /api/debug/version（确认运行版本/环境、httpx依赖版本）

GET /api/debug/ion 等瓦片代理排障接口

3.2 研发效率（TDD）

本地一条命令跑“快速测试基线”：make docker-dev-check。

对外行为（API 契约）用 pytest contract 用例守住：瓦片代理链路、httpx 超时策略、X-AEF-Request-Id 透传等。

前端行为用 vitest 守住：底图门禁/默认行为、HUD 布局、API contract 之关键断言。

3.3 发布可回滚

保持现行 release 软链回滚机制。

引入 Docker 时，优先以“Dev 容器 / Prod 容器 / Canary 容器”增量引入，避免端口与宿主其它服务冲突。

4. Docker 引入策略与核心物料 (Phase 1 落地标准)

目标：让团队在不同机器上拿到一致的 Node/Python 依赖，同时完全兼容 Dev(8404/8405) 与 Prod(8406/8407) 的端口约定和 VITE_API_BASE。

4.1 Dev 环境物料定义（已落地）

docker-compose.dev.yml: 提供 backend, frontend, 以及完全隔离的 backend_test, frontend_test 服务。

.dockerignore: 降低构建上下文成本，加速镜像打包。

Dockerfile.pytest / Dockerfile.test: 专用于 TDD 自动化验证，复制仓库运行测试，不污染宿主 .env。

4.2 前端地图视觉一致性（已落地）

为解决“首页宏观视角/场景微观视角看起来像两张不同地图”的体验差异，前端对 Photorealistic 3D Tiles 做了两级优先级控制：

1) **AI 业务图层优先**：AI 图层激活时，可自动隐藏/变淡 photorealistic tiles（避免遮挡业务栅格）。

2) **相机高度 LOD**：当相机高度高于阈值（默认 2000km）时自动隐藏 photorealistic tiles，避免远景只显示粗糙根瓦片导致“马赛克/破布感”。

常用开关（见 `.env.example`）：

- `VITE_BASEMAP=google_xyz | google_official | osm | grid`
- `VITE_ION_PHOTOREALISTIC_ASSET_ID=...`
- `VITE_PHOTOREALISTIC_VISIBILITY_THRESHOLD_M=2000000`
- `VITE_PHOTOREALISTIC_AI_OCCLUSION=hide|dim|none`

5. TDD 驱动的工作流与防回退机制

为了防止之前 httpx.Timeout 导致 500 崩溃的幽灵 Bug 重现，我们必须以 TDD 的方式为其建立测试基线。

# 2. 一键自动化验收 (推荐！执行 smoke + pytest + vitest，全绿通过)
